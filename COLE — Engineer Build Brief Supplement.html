The COLE Universe — Scaling Vision

The engineer needs to understand where COLE scales so architecture decisions made today don't become blockers tomorrow. Build for what COLE is becoming, not just what COLE is today.

The Core Insight

COLE is not a contact center tool. COLE is an intelligent workforce layer that can sit on top of any business function and run it. The contact center is the first vertical because it's the founder's domain. The architecture must be modular enough to clone into other verticals without rebuilding the core.

Architecture Requirement for Vertical Expansion

The knowledge base, prompt library, and domain rules must be stored as configuration — not hardcoded. When the engineer builds COLE's Brain, it must accept a domain_config.json that defines: industry vocabulary, key metrics, integration targets, escalation paths, and regulatory constraints. Swapping this config = new vertical. This is the modularity that makes COLE a platform, not a product.

THE PHASE 4 NORTH STAR — COLE LMS

Thomas and Albert have reviewed and approved this as the 12-18 month vision. Engineers should be aware of it so today's architecture doesn't block it.

What COLE LMS Does
The 7-step autonomous loop:
1. Monitor — watches real agent performance data (AHT, adherence, quality scores)
2. Identify — detects individual skill gaps per agent
3. Generate — creates personalized micro-training content
4. Deploy — pushes training to the agent automatically
5. Review — scores training completion and comprehension
6. Report — delivers manager briefing: "Here's what happened overnight. Here's what COLE did."
7. Loop — repeats continuously, improving with each cycle
Architecture Prerequisites
These must be built correctly today to enable COLE LMS later:

• Agent performance data ingestion pipeline (Calabrio adapter is the seed)
• Content generation framework with quality scoring
• Multi-tenant data isolation — employee performance data is sensitive
• Modular knowledge base — LMS content is another knowledge config
• The feedback loop architecture — LMS is the feedback loop at scale

Flag: Data privacy and compliance must be designed in at Phase 3 — not retrofitted. Healthcare, finance, and EU clients have regulatory requirements around employee performance data.

Section 4

Build Decisions — Not in Main Brief

These decisions were made during factory planning sessions and must be honored in the build. They are not in the main COLE Build Brief v1.0.

Decision 1 — Database: PostgreSQL from Day One
Decision: PostgreSQL — not SQLite.
Reason: COLE builds profile data, session history, ROAT™ tracking, and team rosters for each client. SQLite hits limits fast. With white-label and multi-tenant Enterprise on the roadmap, PostgreSQL is the only viable choice. Do not start with SQLite and plan to migrate — build PostgreSQL from day one.
Owner confirmation: Mike approved this decision explicitly.
Decision 2 — ROAT™ Dashboard: Web Dashboard (Not CLI)
Decision: Build a web dashboard for ROAT™ health scores — not a CLI report.
Reason: ROAT™ is a marketing asset and a sales tool. Buyers need to SEE their score improving on a screen they can screenshot and share. A CLI report doesn't have the same impact. The dashboard is part of the product experience — not just a reporting utility.
Minimum viable dashboard: Session score over time (line chart), current health score (0-100 gauge), top improvement this week, most-used COLE capability, weekly comparison. React frontend preferred.
Owner confirmation: Mike approved this decision explicitly.
Decision 3 — Engineer Count: Request 3-4 Engineers
Decision: Request 3-4 engineers from CTO.new — not 2.
Reason: 2 engineers = 6 weeks to OAI Enterprise. 3-4 engineers = 3-4 weeks. Given the market timing opportunity (MCP is new, contact center AI space is wide open, no competitors with this architecture yet), speed is a scoring variable. The faster COLE is in market, the larger the first-mover advantage.
Owner confirmation: Mike approved this decision explicitly.
Decision 4 — MCP First Integration: Slack
Decision: Slack is the first MCP integration — before Calabrio, Notion, or HubSpot.
Reason: Every contact center manager already lives in Slack or Teams. COLE in Slack = real-time intelligence where managers already work. Service level alert at 8am in Slack is more valuable than an email. CTO.new confirmed this doubles product appeal. Build Slack MCP at Tier 3 (OAI Architect).
Second MCP: Calabrio — our founder co-led its implementation and has deep technical familiarity. Build Calabrio adapter as a concierge setup for Enterprise clients.
Decision 5 — LLM: Claude Primary, GPT-4 Fallback
Decision: Default LLM is Claude (Anthropic). GPT-4 as fallback. Configurable per client at Enterprise tier.
Reason: 92eleven is built on Claude. The product experience, the prompt architecture, and the brand voice have all been developed in Claude. COLE's personality is most consistent on Claude. GPT-4 fallback ensures reliability. Enterprise clients may have existing OpenAI relationships — configurable LLM is a sales feature at that tier.
Decision 6 — Email Infrastructure: Brevo SMTP
Decision: All COLE-generated emails send from mike@92eleven.com via Brevo SMTP — already authenticated.
Credentials needed: Brevo API key (Mike will provide). Do not use a generic transactional email service — the 92eleven domain authentication is already complete and must be preserved.
Email types COLE sends: Weekly ROAT™ brief, coaching session summaries, director reports, schedule change notifications, follow-up sequences, onboarding confirmations.
Decision 7 — Pricing Architecture: All Prices End in 9
Decision: Every 92eleven price ends in 9. No exceptions. $49 · $99 · $199 · $299 · $499.
Reason: Psychological pricing standard. Consistent across all products. If any pricing UI is built into the platform (checkout, upgrade prompts, plan selection), this rule must be honored in every displayed price.
Decision 8 — COLE's Name Appears at Every Tier
Decision: COLE is the character present at every tier — growing in capability but always COLE.
Implication for UI: Every tier's interface, every email, every dashboard shows COLE's name and references his character. The "COLE office" illustration (navy suit, burnt orange shirt, leaning on desk, 92eleven mug) is his avatar image. At Enterprise tier, voice interactions use "Hi, I'm COLE" as the opening line.
Do not: Use generic "AI assistant" language anywhere in the product. COLE has a name. Use it.
Decision 9 — ROAT™ Formula (Locked)
Decision: The ROAT™ health score formula is locked and must be implemented exactly as specified.
ROAT_score = ( quality_score × 0.40 # Session rating average (1-10 normalized to 0-1) improvement_rate × 0.25 # Week-over-week quality score delta complexity_escalation × 0.20 # Prompt complexity growth rate over sessions satisfaction_trend × 0.15 # Rolling 4-week satisfaction direction ) # Output: 0-100 score. Store per session. Display weekly trend.
Marketing hook tied to this formula: "Are you measuring your ROAT™?" — this is a registered-pending trademark of 92eleven. Always display as ROAT™ in any UI.
Decision 10 — The 103 Prompt Architecture is a Brand Signature
Decision: The COLE System contains exactly 103 prompts — 92 standard + 11 SUPER. This number is intentional and must never be changed without founder approval.
Why: 103 = 92 + 11 = a deliberate reference to 9211 Adams Street — the founder's childhood home and the origin of the 92eleven brand name. This is a brand signature baked into the product architecture. If any UI displays prompt counts, it must show 103 total (92 + 11 SUPER).
SUPER prompt standard: "Would an experienced contact center director pay $49 just for these 11 prompts alone?" If the answer is yes — it's a SUPER prompt. This standard governs which prompts earn the SUPER designation.
